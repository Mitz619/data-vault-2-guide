# Satellite Rules — 6.0

Governing rules for designing and loading Satellites.

---

## 6.1 — Must Import One Parent Hub or Link Key

A Satellite cannot generate its own primary key. It must borrow the PK from its one and only parent Hub or Link.

| Valid | Invalid |
|-------|---------|
| `hub_customer_hk` + `load_dts` as composite PK | A `satellite_sequence_id` auto-generated by the Satellite itself |

## 6.2 — One Parent Only — Never Snowflaked

A Satellite can only attach to ONE parent Hub or Link. It can never chain to another Satellite.

## 6.3 — Load Date Time Stamp is Part of Satellite PK

Unlike Hubs and Links where `LOAD_DTS` is just an attribute, in a Satellite the load date IS part of the primary key. This allows multiple historical versions to coexist.

| PK | Identifies |
|----|-----------|
| `(HK-a3f9, 2024-01-01 08:00)` | Version loaded Jan 1 |
| `(HK-a3f9, 2024-06-01 08:00)` | Updated version loaded Jun 1 — email changed |

Both rows coexist simultaneously. Full history is preserved.

## 6.4 — Sub-Sequence is Optional

For multiple rows arriving at the exact same microsecond (real-time feeds, multi-active satellites), a sub-sequence identifier is added to the PK to guarantee uniqueness.

## 6.5 — Record Source is Required

Every Satellite row must record which source system provided the data.

## 6.6 — Must Have at Least One Descriptive Element

A Satellite containing ONLY the parent key, load date, and record source provides no useful context and is not valid.

## 6.7 — Business Vault Satellites Can Contain Computed Attributes

In the Business Vault layer, a Satellite can contain derived/calculated attributes from soft business rules.

| Layer | Example |
|-------|---------|
| Raw Vault `SAT_` | Raw email exactly as received from source |
| Business Vault `BSAT_` | Validated email, E.164-formatted phone, customer lifetime value |

## 6.8 — Satellites Store Data Over Time — No Physical Updates Ever

| Business Event | What Happens |
|----------------|-------------|
| Customer changes email | New row inserted — old row untouched |
| Customer changes email again | Another new row inserted — all three versions coexist |
| Customer deleted in source | New row in Effectivity Satellite with deleted flag — descriptive Satellite rows untouched |

## 6.9 — Reference Table Access via Natural Key Only

No physical FK constraints to reference tables. Lookups are implied and resolved at query time.

## 6.10 — Satellites Can Be Split or Merged at Any Time

Restructure freely — as long as NO historical data and NO audit trail is lost.

| Action | When | Condition |
|--------|------|-----------|
| Split | Fast and slow attributes causing row proliferation | ALL historical rows migrated before original dropped |
| Merge | Consolidating two sources into one | ALL historical rows from both sources preserved |

---

## Split by Rate of Change — The Key Design Decision

Mixing fast-changing and slow-changing attributes in one Satellite wastes storage and slows delta detection:

```
❌ SAT_CUSTOMER_ALL  (one satellite)
   — email changes weekly → new row every week
   — date_of_birth never changes → duplicated in every weekly row

✅ SAT_CUSTOMER_CONTACT     ← email, phone, loyalty_tier (changes monthly)
✅ SAT_CUSTOMER_DEMOGRAPHICS ← date_of_birth, preferred_name (changes almost never)
```
